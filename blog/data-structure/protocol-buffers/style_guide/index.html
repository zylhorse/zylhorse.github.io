Protocol Buffers - 编码规范

2021-10-28

zylhorse

5分钟

这篇文章讲述.proto文件的编写规范。根据以下约定，将会使你的protocol buffer messgae定义和它们对应的类保持一致并易于阅读。

注意，protocol buffer规范是随着时间改变的，因此你可能会看到用不同的约定或格式编写的.proto文件。 当你修改这些文件的时候，请尊重现有的格式。保持一致就好。但是当你创建新的.proto文件时，最好采用当前的格式。

标准文件格式

Standard file formatting

  • 保持每一行长度限制在80字符内
  • 使用2个空格缩进
  • 字符串最好使用双引号

文件结构

File structure

文件应该命名为: lower_snake_case.proto

所有的文件应该按照以下习惯排列：

  1. License header
  2. File overview(文件概述)
  3. Syntax
  4. Package
  5. Imports (需要排序)
  6. File options(文件选项)
  7. Everything else (其它声明)

Packages

package名称应该时小写，并且应该对应目录层级。e.g.,如果文件在my/package中，那么package名字应该是my.package。

Message和字段名

Message and field names

message: 使用驼峰命名法(首字母大写),例如: SongServerRequest.
字段: 使用underscore_separated_names命名(包括oneof和extension),例如: song_name。 Use CamelCase (with an initial capital) for message names – for example, SongServerRequest. Use underscore_separated_names for field names (including oneof field and extension names) – for example, song_name.

message SongServerRequest {
  required string song_name = 1;
}

对字段名使用这种命名约定，会得到以下访问器:

C++:
  const string& song_name() { ... }
  void set_song_name(const string& x) { ... }

Java:
  public String getSongName() { ... }
  public Builder setSongName(String v) { ... }

Go:
SongServerRequest.SongName = ...
func (x *SongServerRequest) GetSongName() string {...}

如果你的字段名包含数字，数字应该跟在字母后面而不是下划线后面。e.g.,使用song_name1而不是song_name_1。

Repeated字段

Repeated fields

repeated字段使用复数名称:

  repeated string keys = 1;
  ...
  repeated MyMessage accounts = 17;

枚举

Enums

使用驼峰(首字母大写)命名enum类型名称
使用CAPITALS_WITH_UNDERSCORES命名成员名:

enum FooBar {
  FOO_BAR_UNSPECIFIED = 0;
  FOO_BAR_FIRST_VALUE = 1;
  FOO_BAR_SECOND_VALUE = 2;
}

每个enum值后面以分号;结尾，而不是逗号,。 首选前缀命名，而不是将enum包含在message中。 0值应该拼接后缀UNSPECIFIED。

服务

Services

如果你的.proto文件定义了RPC service，你应该使用驼峰(首字母大写)命名service名称和RPC方法名。

service FooService {
  rpc GetSomething(FooRequest) returns (FooResponse);
}

  • Required字段 (仅在proto2中使用)
  • Groups (尽在proto2中使用)
